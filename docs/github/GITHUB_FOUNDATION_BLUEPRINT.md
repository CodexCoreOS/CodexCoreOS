# GitHub Foundation Blueprint

## Purpose

This document defines the top-level GitHub posture for Codex Core as a governed research-and-systems institution with a flagship product, rather than a loose collection of experimental repositories.

The repository-level goal is to make three things legible:

1. doctrine is real
2. continuity is the thesis
3. the companion platform is the current flagship implementation surface

## Recommended Organization Shape

### Institutional root
- Codex Core / CodexCoreOS
- public identity centered on continuity, memory, reasoning, doctrine, and bounded human-centered system design

### Core public repositories
- `.github`
- `Codex-Companion-Platform`
- `codex-core-os`
- `codex-doctrine`
- `codex-archive`
- `codex-docs`
- `codexcore-org-site`

### Private implementation lanes
- internal runtime
- safety evaluations
- memory fabric implementation
- orchestration and product ops

## Repository Role of Codex-Companion-Platform

This repository is the current flagship build surface for the continuity-first companion platform.

It should communicate:
- the current bounded MVP truth
- the doctrine that constrains implementation
- the roadmap from continuity proof to broader platform realization
- how contributors should work without diluting continuity-first identity

## Repository Readability Priorities

### 1. Core message
The system exists to help a user recover the thread of what they were doing when continuity is reduced or lost.

### 2. Visible governance
The repository should make doctrine, contribution pathways, and trust boundaries easy to find.

### 3. Controlled contribution model
Changes should preserve:
- raw before meaning
- ledger before memory
- continuity first
- provenance everywhere
- capacity-aware behavior

## Recommended Repository Standards

Every core Codex repository should eventually include:
- `README.md`
- `LICENSE`
- `CONTRIBUTING.md`
- `SECURITY.md`
- `SUPPORT.md`
- `CODEOWNERS`
- issue templates
- pull request template
- architecture and governance docs

## Immediate Next Steps

1. add default community files
2. add issue templates for bugs, doctrines, and protocol proposals
3. add pull request template aligned to continuity-first review
4. add CODEOWNERS when team map is finalized
5. stand up a public `.github` repository for organization-wide defaults
6. split doctrine, archive, and OS concerns into dedicated repositories when ready

## Design Principle

Design GitHub like a governed research-and-systems institution with a flagship product.
